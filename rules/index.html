VulHunt Rules

VulHunt rules are Lua scripts that encode vulnerability detection logic and run against binary targets. Each rule combines metadata, analysis scopes, and a check function that returns structured findings with evidence annotations. Rules can detect known CVEs, vulnerability patterns, patches, and even previously unknown issues across POSIX binaries and UEFI firmware.

Looking for rules to get started?

Check out our collection of community rules for ready-to-use detection logic covering real-world CVEs and vulnerability patterns.

Rule Structure

A rule is a collection of fields organized into three categories: metadata (author, name), filtering (platform, architecture, conditions), and detection (scopes, types, signatures, extensions). Scopes contain the check functions that implement the actual detection logic.

author = "Binarly"
name = "CVE-2024-12084"
platform = "posix-binary"
architecture = "*:*:*"
types = "rsync/v3.3.0/rsync"
signatures = {project = "rsync", from = "3.2.7", to = "3.3.9"}
conditions = {name_with_prefix = {"rsync"}}

scopes = scope:functions{
  target = {matching = "send_files", kind = "symbol"},
  with = check
}

function check(project, context)
  -- detection logic here
end

Rule Fields

FieldDescriptionCategoryRequired
authorAuthor of the ruleMetadataYes
nameName of the rule (e.g., a CVE identifier)MetadataYes
platformTarget platform: posix-binary, efi-module, or efi-standaloneFilteringYes
architectureArchitecture triplet (processor:endian:bits), e.g., X86:LE:64 or *:*:* for allFilteringYes
conditionsConditions under which the rule should be executedFilteringNo
typesType library used by the engine for matching and decompilationDetectionNo
signaturesSignature files used to identify functions in binaries without symbolsDetectionNo
extensionsSpecifies any extension needed by this rule (e.g., the decompiler)DetectionNo
scopesDefines the list of scopes used to identify a vulnerabilityDetectionYes

Analysis Scopes

Scopes define the analysis context in which detection logic executes. VulHunt supports three scopes, each suited to different detection strategies:

ScopeContextDescription
scope:projectWhole binaryRuns checks across the entire binary. Useful for byte pattern matching and binary-wide analysis.
scope:functionsIndividual functionsTargets specific functions by name, symbol, or query. Ideal for analyzing known vulnerable code.
scope:callsCall-sites in functionsTracks data flow from sources to sinks using taint analysis. The most powerful scope for detecting injection and overflow patterns.

Project Scope

Perform binary-wide analysis such as byte pattern searches:

scopes = {scope:project{with = check}}

Functions Scope

Find and analyze specific functions by name or pattern:

scopes = scope:functions{
  target = {matching = "send_files", kind = "symbol"},
  with = check
}

Calls Scope

Track how attacker-controlled data flows from sources to sinks:

scopes = scope:calls{
  with = check,
  to = {
    matching = "strcpy|strcat|system|sprintf",
    kind = "symbol"
  },
  where = caller:calls "json_object_get_string",
  using = {
    callees = {
      json_object_get_string = {output = var:named "out"}
    }
  }
}

For detailed scope documentation, see the User Guide and Reference.

Detection Capabilities

The engine exposes the following analysis capabilities to rules:

  • Dataflow analysis: trace how values propagate through a function, from attacker-controlled inputs to dangerous sinks. Supports custom sanitization functions to reduce false positives.
  • Code pattern matching: search decompiled code using Weggli-style queries for architecture-independent detection of high-level vulnerability patterns.
  • Type libraries: apply C type definitions and function prototypes to improve decompilation accuracy, enabling detection based on structure fields and function signatures.
  • Function signatures: use FLIRT signatures to restore function names in stripped binaries, making detection logic portable portable across builds with or without symbols.
  • Byte pattern matching: match raw byte sequences in binary code, compatible with FwHunt patterns. Useful for targeting specific instruction sequences or malware indicators.
